리소스 격리 (Resource Quota)
- 하나의 클러스터 안에서 영역을 나누지 않으면 사고가 날 수 있습니다.

- **상황:** Jenkins에서 빌드 작업(DevOps 영역)이 폭주해서 CPU를 100% 다 써버립니다.
- **결과:** 영역 구분이 없다면 옆에 있던 **포털(Stateless)**이나 **DB(Stateful)**가 자원을 뺏겨서 서비스가 느려지거나 멈춥니다.
- **해결:** `Namespace`라는 논리적 구역을 나누고, **"DevOps 구역은 전체 CPU의 20% 이상 못 쓴다"**라고 딱 잘라 제한(Quota)을 걸 수 있습니다.



Namespace로 Resource Quota 호스트의 리소스 사용량의 제한을 두어 서버의 영향도를 주지않도로 설정한다.


## 4. 보안 및 네트워크 격리 (Network Policy)

보안 관점에서도 영역 구분은 필수입니다.
- **Stateless:** 외부 인터넷에서 접속이 가능해야 합니다. (Public)
- **Stateful:** 외부에서 DB에 직접 접속하면 큰일 납니다. 오직 **포털 영역에서 오는 요청만 받도록** 방화벽(Network Policy)을 칠 수 있습니다.

IT 아키텍처에서는 데이터를 어떻게 다루느냐에 따라 **Stateless**와 **Stateful**로 나눕니다. 여기에 관리 도구들을 모은 **DevOps** 영역을 추가하여 설계하는 것이 일반적입니다.

### ① Stateless (상태 비저장) 영역

- **정의:** 데이터를 내부에 저장하지 않는 서비스입니다. 요청이 들어오면 처리하고 끝내며, 서버가 죽었다 살아나도 이전의 기억(데이터)이 필요 없습니다.
- **대상:** 포털 웹 서버, API 서버, Nginx 등.
- **특징:** 언제든 죽이고 새로 띄워도 상관없기 때문에 **확장(Scaling)**과 **무중단 배포**가 매우 쉽습니다.

### ② Stateful (상태 저장) 영역

- **정의:** 서비스가 유지되기 위해 반드시 기억해야 하는 데이터(상태)가 있는 서비스입니다.
- **대상:** DB(MySQL), Redis, 파일 저장소 등.
- **특징:** 서버가 죽었을 때 동일한 데이터(Disk)를 물고 살아나야 합니다. 관리 난이도가 높고, 단순히 개수를 늘린다고 성능이 바로 좋아지지 않습니다.

### ③ DevOps (관리 및 자동화) 영역

- **정의:** 서비스 운영을 지원하는 도구들의 집합입니다.
- **대상:** Jenkins(빌드), GitLab(소스 관리), Prometheus(모니터링) 등.
- **특징:** 서비스 자체

## 2. 가용성(Availability) vs 자동 복구(Self-healing)

이 둘은 바늘과 실 같은 관계지만, 개념적으로는 명확히 다릅니다.

### 가용성 (Availability)

- **개념:** "서비스가 중단되지 않고 계속 운영될 수 있는 능력"입니다.
- **핵심:** **이중화(Redundancy)**입니다.
- **비유:** 자동차 타이어 4개 중 하나가 펑크 나도 보조 타이어 덕분에 계속 달릴 수 있는 상태를 말합니다. "서버가 2대 떠 있어서 1대가 죽어도 사용자는 접속이 된다"면 가용성이 높은 것입니다.

### 자동 복구 (Self-healing)

- **개념:** "문제가 발생했을 때 사람의 개입 없이 스스로 정상 상태로 돌아오는 능력"입니다.
- **핵심:** **복구 프로세스(Recovery)**입니다.
- **비유:** 타이어 펑크를 감지하고, 차가 알아서 갓길에 세운 뒤 새 타이어로 교체하고 다시 출발하는 행위입니다. 쿠버네티스에서 포드가 죽으면 즉시 새 포드를 띄우는 것이 자동 복구입니다.

## 1. 정석 아키텍처: 3-Master & N-Worker

정석 구성의 핵심은 **마스터 노드 3대**를 확보하여 관리 계층의 '절대 중단'을 막는 것입니다.

### ① 구성 요소 (최소 추천: 총 6대)

- **Master Node (3대):** 클러스터의 상태를 관리하고 `etcd` 데이터를 동기화합니다.
- **Worker Node (3대):** 실제 포털, DB, Redis, Jenkins가 돌아가는 공간입니다.
- **Load Balancer (2대 또는 VIP):** 마스터 3대 앞단에서 트래픽을 분산하고, 워커 노드의 Ingress로 연결해주는 입구 역할을 합니다. (Haproxy + Keepalived 조합을 많이 씁니다.)

## 2. 이 구조에서 얻는 이점 (왜 정석인가?)

1. **마스터 노드 장애 대응:** 마스터 1대가 하드웨어 고장으로 죽어도, 남은 2대가 즉시 리더를 선출하여 클러스터 운영을 지속합니다.
2. **워커 노드 장애 대응:** 워커 노드 1대가 죽으면, 그 위에 있던 **포털(Stateless)**은 즉시 다른 워커 노드로 이사 갑니다. **DB(Stateful)** 역시 미리 설정한 공유 스토리지가 있다면 다른 노드에서 데이터를 물고 다시 살아납니다.
3. **작업의 분리:** `DevOps` 영역(Jenkins 빌드 등)이 자원을 엄청나게 잡아먹어도, 마스터 노드의 관리 기능에는 전혀 지장을 주지 않습니다.

---
## 3. 정석대로 구축할 때의 '진짜' 다음 단계

이제 서버 대수는 정해졌으니, 실제 **설계 도면**을 채워야 합니다.

### A. IP 및 Hostname 설계

- 각 서버의 고정 IP를 할당하고, `k8s-master-01`, `k8s-worker-01` 식으로 이름을 정합니다.

### B. 로드 밸런서(LB) 결정

- 마스터 3대를 하나로 묶어줄 주소(Virtual IP)가 필요합니다. 하드웨어 L4 장비가 없다면 소프트웨어로 구성해야 합니다.

### C. 공유 스토리지 선정 (DB 이중화의 핵심)

- **가장 중요한 포인트입니다.** 서버가 여러 대이므로, DB 포드가 어느 노드에 뜨더라도 같은 데이터를 볼 수 있게 해주는 **외부 저장소**가 있어야 합니다. (예: NFS, Ceph, 또는 클라우드라면 AWS EBS/EFS 등)

### D. 런타임 및 네트워크 플러그인(CNI) 선택

- 컨테이너 엔진은 보통 **containerd**를 쓰고, 포드끼리 통신하게 해주는 네트워크망(CNI)은 **Calico**나 **Cilium**을 주로 선택합니다.

## 1. Red Hat 환경에서의 주요 설정 (OS 레벨)

Red Hat은 보안이 엄격하기 때문에 설치 전 다음 사항들을 먼저 조정해야 합니다.

- **SELinux 설정:** 쿠버네티스 포드가 노드의 파일 시스템에 접근할 수 있도록 `Permissive` 모드로 변경하거나 적절한 정책을 적용해야 합니다.

- **Swap 비활성화:** 쿠버네티스는 노드의 메모리 관리를 위해 Swap이 꺼져 있는 것을 기본 전제로 합니다. (`swapoff -a`)


## 4. 해결해야 할 마지막 퍼즐: 공유 스토리지

정석대로 서버 6대를 구성하더라도, **"DB 데이터가 저장되는 물리적 장소"**가 해결되지 않으면 무용지물입니다.

- **NFS 서버 활용:** 별도의 NAS가 있거나 별도 서버에 NFS를 구성하여 모든 워커 노드가 같은 저장소를 바라보게 합니다.
- **OpenEBS / Rook-Ceph:** 별도의 저장 장치가 없다면 워커 노드들의 남는 디스크 공간을 하나로 묶어 가상 스토리지 풀을 만드는 솔루션을 설치합니다.


NFS 서버가 활용가능한지, OpenEBS나 Rook-Ceph 같은 가상 스토리지가 사용 가능한지 확인한다.

## 1. 쿠버네티스 기본: 롤링 업데이트 (Rolling Update)

새 버전을 배포할 때 한 번에 다 바꾸는 게 아니라, **하나씩 순차적으로** 교체하는 방식입니다.

1. **새 Pod 생성:** 새로운 버전(v2)의 포드를 하나 먼저 띄웁니다.
2. **상태 확인 (Health Check):** v2 포드가 정상적으로 준비되었는지 확인합니다.
3. **이전 Pod 삭제:** v2가 준비 완료되면, 기존 버전(v1) 포드 하나를 죽입니다.
4. **반복:** 이 과정을 반복하여 모든 포드를 v2로 교체합니다.

> **핵심:** 이 과정 동안 서비스(Service)는 항상 최소 1개 이상의 'Ready' 상태인 포드로 트래픽을 보내기 때문에 사용자는 끊김을 느끼지 못합니다.

### 1. 왜 3개가 될까요? (MaxSurge 설정)

쿠버네티스 배포 설정(Deployment)에는 `maxSurge`라는 옵션이 있습니다.

- **maxSurge:** "배포 중에 설정된 리플리카(Replica) 수보다 얼마나 더 많은 포드를 동시에 띄울 수 있는가?"를 결정합니다.
- 기본값이 보통 **25%** 또는 **1**로 설정되어 있어서, 리플리카가 2개라면 새 버전 포드 1개를 먼저 추가로 띄워 총 **3개**가 됩니다.

---
## 1. Ingress Controller란 무엇인가?

쉽게 비유하자면, Ingress Controller는 **"클러스터 전용 스마트 호텔 리셉션"**입니다.


### 왜 그냥 Nginx 서버를 따로 안 쓰고 이걸 쓰나요?

- **동적 업데이트:** 일반 Nginx는 포드가 죽고 새로 생겨서 IP가 바뀔 때마다 사람이 설정 파일을 수정하고 재시작해야 합니다. 하지만 Ingress Controller는 포드의 변화를 실시간으로 감지해서 **자동으로 경로를 수정**합니다.
    
- **L7 기능 통합:** 하나의 입구(IP)로 들어와서 도메인 이름(`portal.com` vs `api.com`)이나 경로(`portal.com/login`)에 따라 서로 다른 포드로 쏴주는 **기능(Routing)**과 **SSL 인증서(HTTPS)** 처리를 한 곳에서 다 합니다.
    
- **표준화:** YAML 파일 하나로 "이 도메인은 이 서비스로 연결해"라고 정의할 수 있어 관리가 매우 표준화됩니다.


---

포드 10~20개는 쿠버네티스 세계에서는 사실 **'매우 작은(Lightweight)'** 규모에 속합니다. 이때 K3s를 고민하시는 건 아주 합리적인 생각입니다.

### K3s란?

Rancher에서 만든 **'경량화된 쿠버네티스'**입니다. 정식 K8s에서 잘 쓰이지 않는 기능을 제거하고, 실행 파일을 단 하나로 합쳐서 메모리 사용량을 대폭 줄였습니다.


| **비교 항목**   | **K8s (정석)**                | **K3s (경량)**                                    |
| ----------- | --------------------------- | ----------------------------------------------- |
| **리소스 점유**  | 마스터 노드 권장 사양이 높음 (RAM 4GB+) | 매우 낮음 (RAM 512MB~1GB로도 구동 가능)                   |
| **설치 및 관리** | `kubeadm` 등 설정할 게 많고 복잡함    | 바이너리 하나로 설치 끝, 관리가 매우 쉬움                        |
| **내장 컴포넌트** | 별도 설치 필요 (etcd, ingress 등)  | **etcd 대신 SQLite 선택 가능, 배포판에 인그레스(Traefik) 포함** |
| **안정성**     | 대규모 클러스터(노드 수백 대) 검증 완료     | 소규모, Edge 컴퓨팅, 단일 서버 환경에 최적화                    |


사용자님의 상황(**Red Hat 서버 6대, 정석 배포, 포드 10~20개**)이라면 다음과 같이 결정하시는 것이 좋습니다.

1. **"나는 인프라 관리에 쏟을 시간이 적고, 쉽고 빠르게 가고 싶다"** → **K3s 추천**
    - 6대의 사양이 낮아도 쌩쌩 돌아갑니다. 포드 20개 정도면 K3s가 훨씬 쾌적할 수 있습니다.

Nginx Ingress Controller (K8s)


## 1. "Nginx 컨테이너" vs "Ingress-Nginx"

사용자님이 말씀하신 **'Nginx 컨테이너'**는 우리가 흔히 `docker pull nginx` 해서 쓰는 그 순수한 웹 서버 소프트웨어입니다. 반면, **`kubernetes/ingress-nginx`**는 그 Nginx 컨테이너를 쿠버네티스용으로 개조하고 **'똑똑한 관리자(Controller)'**를 붙여놓은 패키지 상품입니다.

- **그냥 Nginx 컨테이너를 쓰면?** 포드(Pod)가 새로 뜰 때마다 사용자가 수동으로 `nginx.conf` 파일을 수정하고 재시작해야 합니다. (사실상 쿠버네티스를 쓰는 의미가 사라집니다.)
- **Ingress-Nginx를 쓰면?** 사용자는 YAML로 규칙만 정의하고, 내부 관리자가 포드의 변화를 감시하며 Nginx 설정을 **실시간으로 자동 업데이트**합니다.

어떤 파드로 변경될지 정해준다.
따라서 쿠버네티스 환경에서는 무조건 **`kubernetes/ingress-nginx`**를 쓰는 것이 정석입니다

**Nginx Ingress**가 일반 Nginx와 다른 결정적인 이유가 바로 그 **'자동 매핑(Dynamic Discovery)'** 기능에 있습니다. 그리고 이것이 L4 로드밸런서와 결정적으로 다른 지점이기도 합니다.

## 2. L4 로드밸런서 vs L7 Ingress (Nginx)

가장 많이 헷갈리는 부분입니다. 전송 계층(Layer)이 다르기 때문에 하는 역할이 명확히 나뉩니다.

| **구분**     | **L4 로드밸런서 (전송 계층)**       | **L7 Ingress/Nginx (응용 계층)**   |
| ---------- | -------------------------- | ------------------------------ |
| **판단 기준**  | **IP 주소와 포트 번호**만 보고 전달    | **URL, 도메인, 쿠키, 헤더** 내용을 보고 전달 |
| **특징**     | 단순히 "들어온 놈을 빈 서버로 보낸다"     | "주소를 보고 어떤 서비스로 보낼지 결정한다"      |
| **비유**     | 아파트 입구의 **차단기** (동/호수만 확인) | 아파트 현관의 **우편함** (이름/내용물 확인)    |
| **K8s 예시** | 클라우드의 NLB, MetalLB         | Nginx Ingress Controller       |

- **L4 (외부 입구):** 외부에서 들어오는 거대한 트래픽을 일단 받아서, 클러스터 내부에 떠 있는 **Nginx Ingress 포드들**에게 골고루 나눠줍니다. (이때는 IP/Port만 봅니다.)
- **L7 (내부 길잡이):** 트래픽을 받은 Nginx Ingress는 이제 패킷을 까봅니다. "아, `portal.com` 주소네? 그럼 포털 포드로 가세요!"라고 세밀하게 길을 안내합니다.

## 1. Nginx Ingress의 자동 매핑 원리

일반 Nginx는 IP 주소를 일일이 적어줘야 하지만, **Nginx Ingress**는 쿠버네티스의 **Service**라는 객체를 바라봅니다.

1. **감시:** Ingress Controller는 쿠버네티스 API를 실시간으로 지켜보며 "포털 포드가 새로 떴나? 죽었나?"를 체크합니다.
2. **엔드포인트(Endpoint) 업데이트:** 포드가 새로 뜨면 그 포드의 **내부 IP**를 알아내서 Nginx 설정(Upstream)에 자동으로 추가합니다.
3. **트래픽 전달:** 사용자가 접속하면, Nginx는 방금 업데이트된 따끈따끈한 IP 리스트 중 하나로 트래픽을 쏴줍니다.

**이 모든 과정이 1초 내외로 자동으로 일어나기 때문에, 우리는 서버 IP가 바뀌어도 설정을 건드릴 필요가 없는 것입니다.**

---


## 1. 추천 구성: 마스터 겸 워커 통합형 (3노드 HA)

서버 6대를 굳이 3:3으로 나눌 필요 없이, **3대만으로도 완벽한 고가용성을 구축**하고 나머지 3대는 대기 자원으로 두거나 다른 용도로 쓰는 방식입니다.

### 역할 배분 (총 3대 사용)

- **Node 1, 2, 3:** 각각 마스터(Server) 역할을 수행하면서, 동시에 워커(Agent) 역할도 수행합니다.
    
- **특징:**
    
    - **고가용성 확보:** 서버 1대가 죽어도 클러스터 관리와 서비스가 모두 살아남습니다.
    - **자원 활용 극대화:** CMP는 트래픽보다 기능 동작 위주이므로, 마스터가 노드 자원을 독점하게 두는 것은 낭비입니다. 마스터 노드에 포털과 DB를 같이 띄웁니다.
    - **관리 포인트 감소:** 서버 6대를 관리하는 것보다 3대를 촘촘하게 관리하는 것이 운영 공수가 적게 듭니다.

## 2. CMP 솔루션 최적화 전략 (K3s 기반)

트래픽이 적은 CMP라면 **K3s**를 쓰는 것이 신의 한 수입니다.

- **DB 및 Redis 통합:** CMP는 보통 관리용 데이터가 핵심입니다. 외부 스토리지를 복잡하게 잡기보다, 3대의 노드에 데이터를 복제하는 **Longhorn** 같은 경량 스토리지 솔루션을 얹어서 사용하세요.
- **Ingress:** 기본 내장된 **Traefik**을 그대로 써도 무방합니다. (Nginx보다 가볍고 CMP 대시보드와 궁합이 좋습니다.)
- **DevOps:** Jenkins 대신 **GitHub Actions Runner**나 **GitLab Runner**를 컨테이너로 띄워 필요할 때만 자원을 쓰게 설정하세요.


## 3. 구축 시 주의할 점 (Taint와 Toleration)

보통 정석 쿠버네티스 설치 도구(kubeadm 등)는 보안과 안정성을 위해 **"마스터 노드에는 일반 포드를 띄우지 마라"**는 표식(**Taint**)을 자동으로 새겨둡니다.

통합형으로 쓰시려면 이 표식을 지워주는 작업이 필요합니다.
- **K3s:** 기본적으로 마스터에 포드가 뜨도록 설정되어 있어 별도 작업이 필요 없습니다. (매우 편리하죠!)
- **K8s (정석):** `kubectl taint nodes --all node-role.kubernetes.io/master-` 같은 명령어로 금지령을 풀어줘야 합니다.


### 1. [옵션 A] 서버 3대 통합형 (All-in-One)

서버 3대에 포털, DB, Jenkins, GitLab 등을 모두 때려넣는 방식입니다.

- **장점:** * **관리 공수가 절반:** 업데이트, 보안 패치, 백업을 서버 3대분만 하면 됩니다
    - **자원 낭비 제로:** 모든 서버의 CPU/RAM을 알뜰하게 나누어 씁니다.
- **단점 (Risk):** * **자원 간섭:** Jenkins가 무거운 빌드를 돌리는 순간, 옆에 있던 포털 사이트가 버벅일 수 있습니다.
    - **확장성 한계:** 나중에 서비스가 커지면 결국 서버를 늘려야 하는데, 그때 가서 분리하려면 머리가 아픕니다.
- **추천 사양:** 서버 1대당 **CPU 8코어 / RAM 16GB~32GB** 이상일 때.
    

---

### 2. [옵션 B] 서버 6대 분리형 (Service + Infra)

운영용 3대와 개발 지원용(CI/CD) 3대로 나누는 방식입니다.

- **장점:** * **장애 격리 (Blast Radius):** GitLab이나 Jenkins가 죽거나 자원을 다 써버려도, 사용자가 쓰는 **포털과 DB는 평온**합니다.
    - **업데이트 용이:** 개발 환경에서 먼저 테스트해보고 운영에 배포하는 정석 프로세스가 물리적으로 완벽히 보장됩니다.
- **단점:** * **관리 포인트 증가:** OS 업데이트와 쿠버네티스 관리를 두 세트(총 6대) 해야 합니다.
- **추천 사양:** 서버 1대당 **CPU 4~8코어 / RAM 8GB~16GB** 정도일 때.

---

### 1. "둘은 절대 같이 두지 마" (Anti-Affinity)

사용자님이 말씀하신 **"포털 Pod끼리는 서로 다른 노드에 찢어놔라"**는 규칙입니다. 서버 1대가 죽어도 다른 서버에 있는 포털이 살아남게 하려면 필수입니다.

- **설정 방식:** `podAntiAffinity`를 사용합니다.
- **논리:** "내 이름이 `portal`인 Pod를 찾아서, 그 Pod가 이미 있는 노드에는 나를 배정하지 마!"라고 선언합니다.


### 💡 실무 적용 시나리오 (3~6대 환경)

사용자님의 CMP 환경에 적용한다면 이런 규칙을 넣게 됩니다.

1. **Portal:** `Anti-Affinity`를 걸어서 노드 1, 2, 3번에 하나씩 찢어서 배치 (서버 장애 대비).
2. **DB:** `Node Selector`를 써서 특정 노드에 고정하거나, 공유 스토리지 성능이 보장된 노드 그룹으로 유도.
3. **Jenkins/GitLab:** `Resource Requests/Limits`를 빡빡하게 걸어서 다른 서비스의 자원을 뺏지 못하게 격리.

### 4. CMP 솔루션에서의 추천 조합

사용자님은 **CMP(Cloud Management Platform) 솔루션**을 구축 중이시니, 인프라 자체가 상품이 될 수도 있겠네요. 그렇다면 이 조합이 가장 정석입니다.

- **관리용 툴:** **Rancher**를 설치하세요. (무료 오픈소스)
- **배포 방식:** **Helm 차트** 형태로 서비스를 패키징하세요.
- **운영:** Rancher UI에서 해당 Helm 차트를 불러와, 마우스 클릭으로 **Node Affinity**나 **Replica** 개수를 조절하며 운영합니다.

### 1단계: K3s 설치 (HA 구성)

- **주의:** 3대 모두 마스터(Server)이므로, 첫 번째 서버 설치 후 나오는 **Token**을 잘 복사해서 나머지 2대를 `server` 모드로 조인해야 합니다.
- **Red Hat 특이사항:** `firewalld`에서 6443(API), 2379~2380(etcd) 포트를 반드시 열어주세요.

### 2단계: Rancher & Helm 설치

- **Rancher 위치:** Rancher는 클러스터 전체를 관리하는 앱이므로, 특정 서버 한 대에 고정하기보다 **쿠버네티스 위의 포드(Pod) 형태**로 올리는 것이 정석입니다. 그래야 Rancher가 떠 있는 서버가 죽어도 다른 서버에서 Rancher UI가 다시 살아납니다.
- **Helm:** Helm은 서버에 무겁게 깔리는 게 아니라 실행 파일 하나(`binary`)이므로 마스터 노드 어디서든 실행 가능하게만 두시면 됩니다.

### 3단계: Namespace 정의

- **이름 추천:**
    - `cmp-stateless` (포털, 프론트)
    - `cmp-stateful` (DB, Redis)
    - `cmp-infra` (Jenkins, GitLab, Guacamole)
        
- **이점:** 나중에 자원 제한(Quota)을 걸 때 "infra 네임스페이스는 전체 메모리의 40% 이상 쓰지 마!"라고 구역별로 제어하기 편해집니다.

### 4단계: 자원 및 배치 세팅 (가장 중요한 설계)

- **Resource Limit:** Jenkins와 GitLab에는 반드시 CPU/Memory **Limit**을 거세요. (안 그러면 빌드 한 번에 포털이 멈출 수 있습니다.)
- **Affinity:** 포털 포드는 최소 2개로 설정(`replicas: 2`)하고, `Anti-Affinity`를 걸어 서로 다른 서버에 뜨게 만드세요.

### 5단계: 배포 및 Failover 테스트

- **테스트 방법:** 포털을 띄워놓고 접속한 상태에서, 서버 한 대의 전원을 강제로 끄거나 랜선을 뽑아보세요.
- **관전 포인트:** 1. 마스터 권한이 다른 노드로 잘 넘어가는가? 2. 죽은 서버에 있던 포드들이 살아있는 서버로 이사 가서 다시 `Running`이 되는가? 3. 접속 중인 화면에서 끊김이 얼마나 발생하는가? (무중단 확인)
