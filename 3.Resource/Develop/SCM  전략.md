## 1. "Merge-Proof" 아키텍처: 내부 SDK화

코어 로직이 변했을 때 에러가 날까 봐 걱정되는 이유는, 고객사 로직이 코어의 **'내부 장기(Internal Logic)'**를 너무 깊숙하게 건드리고 있기 때문입니다.

- **해결책**: 코어는 고객사 로직에게 **"여기까지만 건드려"**라고 선을 긋는 **Interface(또는 Abstract Class)**만 제공해야 합니다.
- **방법**: 코어 안에 `AutomationTask`라는 추상 레이어를 만듭니다.
    
    - 코어가 업데이트되어 내부 DB 구조가 바뀌더라도, `AutomationTask`라는 입구(Interface)의 규격만 유지한다면 SK-Hynix 브랜치의 Jenkins 로직은 절대 깨지지 않습니다.


플러그인이 많아지는 것은 성공의 측면에서 좋다.


## 1. 플러그인 아키텍처의 철학: "개방-폐쇄 원칙 (OCP)"

소프트웨어 설계 원칙 중 **OCP(Open-Closed Principle)**라는 것이 있습니다.

- **Open:** 새로운 기능 추가에는 열려 있어야 하고
- **Closed:** 기존의 코드(코어) 수정에는 닫혀 있어야 한다.
    
즉, **코어를 한 줄도 고치지 않고도** Jenkins 연동, Ansible 연동 같은 기능을 뗐다 붙였다 할 수 있어야 진정한 플러그인 아키텍처라고 할 수 있습니다.

최소 기능 제품(MVP) 단위여야 한다.

Core : 제품의 뼈대 
Plugin : 확장 기능 

|**구분**|**Core (제품의 뼈대)**|**Plugin (확장 기능)**|
|---|---|---|
|**기준**|**이 기능 없이는 포탈 구실을 못함**|**있으면 좋지만, 없어도 시스템은 가동됨**|
|**예시**|인증/인가, 메뉴 구조, DB 연결, 로깅, 이벤트 버스|Jenkins 연동, Ansible 자동화, 특정 솔루션 대시보드|
|**변경 주기**|매우 보수적 (분기별/반기별 업데이트)|매우 유연 (고객사 요청 시 상시 업데이트)|
|**영향도**|수정 시 모든 고객사에 영향|특정 고객사 혹은 해당 기능 사용자에게만 영향|

## . 전략 A: Jenkins를 코어(Standard)로 포함하는 경우

만약 Clovirone 2.0의 시장 경쟁력이 "OS 자동화까지 한 번에 다 된다"는 점이고, 모든 고객사에게 Jenkins 연동을 기본으로 제공하기로 했다면 **코어에 포함**할 수 있습니다.

- **장점:** 설치가 한 번에 끝나고 관리가 단순합니다.
    
- **단점:** 나중에 "우리는 Jenkins 말고 GitLab CI 써요"라는 고객사가 나오면 코어 로직을 또 고쳐야 합니다.
    

## 2. 전략 B: Jenkins를 'Default Plugin'으로 관리하는 경우 (권장)

가장 영리한 방법은 **Jenkins를 코어에 넣지 않되, 제품 패키지에는 기본으로 포함**시키는 것입니다.

- **구조:** * **Core:** 자동화 엔진 인터페이스
    
    - **Plugin:** Jenkins Plugin (기본 탑재)
        
- **효과:** 고객사 입장에서는 Jenkins가 기본으로 제공되는 것처럼 보이지만, 개발자 입장에서는 코어와 분리되어 있어 유지보수가 쉽습니다.


- **Standard Bundle:** [Core] + [Jenkins Plugin] + [Ansible Plugin]
- **Light Bundle:** [Core] 만 포함

## 3. 코어 포함 여부를 결정하는 '최종 체크리스트'

다음 세 가지 질문에 모두 **YES**가 나와야 코어에 넣는 것이 안전합니다.

1. **필수성:** "이 기능(Jenkins)이 없으면 우리 솔루션은 가치가 없는가?"
2. **보편성:** "우리 고객의 90% 이상이 이 도구를 그대로 사용하는가?"
3. **안정성:** "이 도구의 API나 연동 방식이 앞으로 1~2년간은 변하지 않을 것인가?"


### Clovirone 2.0 최종 아키텍처 지도

1. **Core (The Heart)**
    
    - **대상:** Aria Automation 연동, 사용자 권한, 대시보드 프레임, 플러그인 인터페이스.
    - **특징:** 이 기능이 없으면 "Clovirone"이라 부를 수 없음.
        
2. **Default Plugins (The Standard Pack)**
    
    - **대상:** Jenkins, Ansible, GitLab 등 (대부분의 고객이 기본으로 기대하는 기능).
    - **전략:** 코어와 분리된 소스 구조를 가지지만, 제품 출고 시 기본으로 포함됨.
        
3. **Common Plugins (The Library)**
    
    - **대상:** 2~3개 이상의 고객사가 공통으로 요청한 기능 (예: 특정 국산 백업 솔루션 연동 등).
    - **전략:** 필요한 고객사에게만 '선택 옵션'으로 제공.
        
4. **Customer-Specific Plugins (The Custom Tailoring)**
    
    - **대상:** SK-Hynix, POSCO 등 특정 고객사의 독특한 레거시 연동이나 비즈니스 로직.
    - **전략:** **해당 고객사 전용 레포지토리/폴더**에서 관리하여 코어 업데이트 시 간섭 최소화.

**"이 소스코드가 물리적으로 어디에 위치해야 하는가"**에 대한 **설계적 기준(Architectural Standard)**이 없었기 때문입니다.
