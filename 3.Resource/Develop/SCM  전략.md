### DESC 

Clovirone 2.0은 특정 고객사(SK-hynix, POSCO 등)의 과도한 커스터마이징으로 인해 **제품이 아닌 개별 SI 프로젝트**로 파편화되고 있습니다. 기본적인 Git-Flow만으로는 고객사 브랜치와 코어 브랜치 간의 병합(Merge) 시 발생하는 논리적 충돌과 의존성 문제를 해결할 수 없습니다.

### How to Solve Problem ?

문제 해결의 핵심은 소스코드가 물리적으로 어디에 위치해야 하는가에 대한 기준을 세우는 것


### A. 코어(Core) 판별 기준: MVP 전략

- 기준: 이 기능이 없으면 Clovirone 2.0이 동작조차 하지 않는가?
- 제품의 뼈대 (인증, 메뉴, DB 연결, Aria 연동, 플러그인 인터페이스)
- Plugin: 확장 기능 (Jenkins, Ansible, 고객사 전용 후처리 로직


### B. "Merge-Proof" 아키텍처: 내부 SDK화

고객사 로직이 코어의 내부를 건드리지 못하도록 **Interface(추상 레이어)**를 제공합니다. 코어의 내부 구조가 바뀌어도 인터페이스 규격만 유지된다면 고객사 브랜치의 로직은 깨지지 않습니다.


| **구분**               | **대상 기능**                  | **관리 및 배포 전략**                         |
| -------------------- | -------------------------- | -------------------------------------- |
| **Core (The Heart)** | Aria Automation 연동, 인프라 로직 | 제품의 심장. 모든 고객사 공용.                     |
| **Default Plugins**  | Jenkins, Ansible, GitLab   | **전략 B(권장)** 적용. 코어와 분리하되 패키징 시 기본 포함. |
| **Common Plugins**   | 범용적인 외부 솔루션 연동             | 2개 이상 고객사 요청 시 라이브러리화하여 공유.            |
| **Customer Plugins** | 고객사 전용 특화 로직 (SI성)         | **물리적 격리.** 해당 고객사 전용 폴더/레포지토리에서 관리.   |
