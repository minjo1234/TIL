

## **📌 예상 질문 & 답변 예시**


### **1. EC2 인스턴스를 생성할 때 어떤 요소들을 설정해야 하나요?**

  

**답변:**

“EC2 인스턴스 생성 시에는 OS 이미지(AMI), 인스턴스 타입(CPU·메모리), 스토리지(EBS), 네트워크 환경(VPC, Subnet), 보안 그룹, 키 페어를 주로 설정합니다. 이후 태그나 IAM Role을 통해 운영과 권한 관리까지 고려합니다.”

---

### **2. 서버 성능이 저하되었을 때 어떻게 대응하시나요?**

  

**답변:**

“먼저 CloudWatch나 모니터링 툴을 통해 CPU, 메모리, 네트워크 지표를 확인합니다. 애플리케이션 로그와 외부 API 호출 빈도를 분석해 병목을 파악하고, 필요 시 인스턴스 타입 변경이나 Auto Scaling을 적용합니다. 이전에는 Prometheus+Grafana 기반으로 VM/컨테이너 리소스를 추적해 장애를 사전에 감지한 경험이 있습니다.”

---

### **3. AWS 비용 최적화를 위해 어떤 방법을 사용해봤나요?**

  

**답변:**

“스펙이 과할 경우 리사이징을 하고, 사용하지 않는 리소스를 즉시 정리합니다. 장기적으로는 예약 인스턴스(RI)나 스팟 인스턴스를 활용할 수 있습니다. 또 모니터링을 통해 실제 자원 사용량과 요청량을 비교해 과도한 배치를 줄이는 식으로 최적화합니다.”

---

### **4. IAM에서 최소 권한 원칙을 어떻게 적용했나요?**

  

**답변:**

“모든 계정에 Administrator 권한을 주는 대신, 역할별로 필요한 리소스만 접근할 수 있는 정책을 부여했습니다. 예를 들어, DB 운영자는 EC2나 S3에 접근할 필요가 없으므로 RDS 관련 권한만 주는 식입니다. 또 액세스 키는 주기적으로 로테이션하고, MFA를 통해 보안을 강화했습니다.”

---

### **5. 장애가 발생했을 때 어떻게 대응하시나요?**

  

**답변:**

“우선 모니터링 알람을 통해 장애 상황을 빠르게 인지합니다. 서비스 로그와 자원 상태를 점검한 뒤, 장애가 애플리케이션 문제인지 인프라 문제인지 구분합니다. 이전에는 VM 생성 과정에서 병목이 발생한 적이 있었는데, 이벤트 기반 아키텍처로 개선해 장애가 확산되지 않도록 한 경험이 있습니다. AWS에서도 동일하게 서비스와 인프라를 분리해 대응할 수 있다고 생각합니다.”

---

## **📌 1. 기본적인 AWS 자원 이해**

  

**Q. EC2 인스턴스를 생성할 때 어떤 요소들을 설정해야 하나요?**

👉 _“AMI(운영체제), 인스턴스 타입(CPU·메모리), 스토리지(EBS), 네트워크(VPC, Subnet), 보안 그룹, 키 페어를 주로 설정합니다. 태그와 IAM Role까지 고려해 운영과 접근권한을 관리합니다.”_

  

**Q. 보안 그룹(Security Group)과 네트워크 ACL 차이를 설명해주세요.**

👉 _“보안 그룹은 인스턴스 단위의 가상 방화벽으로 상태 기반(Stateful)이고, NACL은 서브넷 단위로 동작하며 Stateless라서 Inbound/Outbound 규칙을 모두 설정해야 합니다.”_

  

**Q. AWS에서 스토리지는 어떤 옵션들이 있고, 언제 사용하는 게 좋은가요? (EBS vs S3 vs EFS)**

👉 _“EBS는 EC2에 붙는 블록 스토리지로 DB 같은 워크로드에 적합합니다. S3는 객체 스토리지로 이미지·백업 파일 저장에 쓰이고, EFS는 여러 인스턴스가 동시에 접근 가능한 네트워크 파일 시스템이라 공유 스토리지 용도에 적합합니다.”_

## **📌 2. 운영 및 관리**

  

**Q. 서버 성능이 저하되었을 때 어떤 방식으로 모니터링하고 원인을 파악하시나요?**

👉 _“CloudWatch로 CPU·메모리·네트워크 지표를 확인하고, 애플리케이션 로그와 외부 API 호출 빈도를 분석합니다. 필요하면 APM 도구를 연계합니다. 이전에는 Prometheus+Grafana로 VM/컨테이너를 추적해 병목을 조기 발견한 경험이 있습니다.”_

  

**Q. EC2에서 장애가 발생했을 때 어떻게 대응하시겠습니까?**

👉 _“우선 CloudWatch 알람과 상태 체크로 장애를 인지하고, 인스턴스 자체 문제인지 애플리케이션 문제인지 구분합니다. 인프라 문제면 인스턴스 교체나 Auto Recovery를, 애플리케이션 문제면 로그 분석 후 롤백 배포를 고려합니다.”_

  

**Q. AWS 비용 최적화를 위해 어떤 방법을 사용해봤나요?**

👉 _“스펙 다운사이징, 사용하지 않는 리소스 종료, 예약 인스턴스와 스팟 인스턴스 활용, 그리고 모니터링 기반으로 실제 사용량과 리소스 크기를 맞추는 방식으로 최적화할 수 있습니다.”_

---

## **📌 3. 배포 및 자동화**

  

**Q. AWS에서 애플리케이션을 배포할 때 어떤 방법을 써보셨나요?**

👉 _“기본적으로는 직접 EC2에 배포했지만, Elastic Beanstalk이나 CodeDeploy를 사용하면 무중단 배포가 용이합니다. 컨테이너 기반이라면 ECS나 EKS도 활용 가능합니다.”_

  

**Q. 오토스케일링 경험이 있으신가요? 있다면 어떻게 설정했나요?**

👉 _“Auto Scaling 그룹을 설정해 CPU 사용률 70% 이상이면 인스턴스를 추가하고, 30% 이하로 떨어지면 줄이는 방식으로 운영했습니다. 로드밸런서와 연계해 트래픽을 자동 분산했습니다.”_

  

**Q. 인프라 환경을 코드(IaC)로 관리해본 경험이 있나요?**

👉 _“Terraform이나 CloudFormation으로 IaC를 구성하면 버전 관리와 재현성이 뛰어납니다. 직접 IaC를 사용한 경험은 많지 않지만, Docker Compose 기반 인프라 관리 경험을 AWS IaC로 확장할 수 있다고 생각합니다.”_

---

## **📌 4. 보안 및 계정 관리**

  

**Q. IAM에서 최소 권한 원칙을 어떻게 적용했는지 설명해주세요.**

👉 _“모든 사용자에게 관리자 권한을 주지 않고, 역할별로 필요한 리소스만 접근하도록 정책을 분리했습니다. 예를 들어 DB 운영자는 RDS 관련 권한만, 개발자는 S3 접근만 허용하는 식입니다.”_

  

**Q. 키 페어(ssh key) 관리를 어떻게 하셨나요?**

👉 _“개인 키는 로컬에서 안전하게 관리하고, 서버에는 공개 키만 배포했습니다. 키는 주기적으로 로테이션하고 필요 없는 키는 즉시 회수했습니다.”_

  

**Q. EC2에 접근 제어를 걸 때 어떤 방식으로 설정했는지 알려주세요.**

👉 _“보안 그룹에서 특정 IP만 SSH 포트(22)를 열었고, Bastion Host를 통해서만 내부 서버에 접근하도록 구성했습니다.”_

## **📌 5. 모니터링 및 성능 최적화**

  

**Q. CloudWatch나 다른 모니터링 도구를 사용해본 경험이 있나요?**

👉 _“CloudWatch 기본 모니터링을 사용할 수 있고, 이전에는 Prometheus+Grafana로 VM/컨테이너 메트릭을 수집·시각화해 실시간 모니터링을 구축한 경험이 있습니다.”_

  

**Q. VM/EC2 자원의 CPU·메모리가 한계에 다다르면 어떤 조치를 하시겠습니까?**

👉 _“먼저 병목 원인을 파악하고, 애플리케이션 최적화를 시도합니다. 그래도 부족하면 인스턴스 타입 업그레이드나 Auto Scaling을 통해 자원을 확장합니다.”_

  

**Q. 서버 성능 테스트를 어떻게 진행했는지 설명해주실 수 있나요?**

👉 _“JMeter나 Locust 같은 부하 테스트 도구로 시뮬레이션을 진행하고, 모니터링 지표와 연계해 병목을 확인합니다. ERP 운영 시에도 테스트 환경에서 사전 검증을 통해 안정성을 확보했습니다.”_

## **📌 6. 경험 기반 질문**

  

**Q. ERP나 Cloud 웹 플랫폼 운영 시, VM 자원 최적화를 위해 어떤 방식을 적용하셨나요?**

👉 _“사용률을 기반으로 VM 스펙을 조정하고, 불필요한 VM은 회수했습니다. 또 스타 스키마 기반 DB 구조 리팩토링과 인덱스 최적화로 조회 성능을 개선해 자원 소모를 줄였습니다.”_

  

**Q. VMware 기반 VM 운영 경험이 AWS 인프라 관리에 어떻게 연결된다고 생각하시나요?**

👉 _“VM 자원 할당과 모니터링 경험은 EC2 운영과 유사합니다. VMware에서 CPU·메모리 최적화를 했던 경험을 AWS 오토스케일링이나 인스턴스 리사이징에 그대로 적용할 수 있다고 생각합니다.”_

  

**Q. Prometheus/Grafana로 모니터링을 구축하셨는데, AWS 환경이라면 어떤 식으로 대체/적용 가능할까요?**

👉 _“AWS에서는 CloudWatch와 CloudWatch Logs로 기본 모니터링을 하고, 필요하면 Grafana와 연동해 시각화를 강화할 수 있습니다. 기존 Prometheus 기반 경험을 CloudWatch 지표에 그대로 확장할 수 있다고 봅니다.”_
