

## **📌 예상 질문 & 답변 예시**


### **1. EC2 인스턴스를 생성할 때 어떤 요소들을 설정해야 하나요?**

  

**답변:**

“EC2 인스턴스 생성 시에는 OS 이미지(AMI), 인스턴스 타입(CPU·메모리), 스토리지(EBS), 네트워크 환경(VPC, Subnet), 보안 그룹, 키 페어를 주로 설정합니다. 이후 태그나 IAM Role을 통해 운영과 권한 관리까지 고려합니다.”

---

### **2. 서버 성능이 저하되었을 때 어떻게 대응하시나요?**

  

**답변:**

“먼저 CloudWatch나 모니터링 툴을 통해 CPU, 메모리, 네트워크 지표를 확인합니다. 애플리케이션 로그와 외부 API 호출 빈도를 분석해 병목을 파악하고, 필요 시 인스턴스 타입 변경이나 Auto Scaling을 적용합니다. 이전에는 Prometheus+Grafana 기반으로 VM/컨테이너 리소스를 추적해 장애를 사전에 감지한 경험이 있습니다.”

---

### **3. AWS 비용 최적화를 위해 어떤 방법을 사용해봤나요?**

  

**답변:**

“스펙이 과할 경우 리사이징을 하고, 사용하지 않는 리소스를 즉시 정리합니다. 장기적으로는 예약 인스턴스(RI)나 스팟 인스턴스를 활용할 수 있습니다. 또 모니터링을 통해 실제 자원 사용량과 요청량을 비교해 과도한 배치를 줄이는 식으로 최적화합니다.”

---

### **4. IAM에서 최소 권한 원칙을 어떻게 적용했나요?**

  

**답변:**

“모든 계정에 Administrator 권한을 주는 대신, 역할별로 필요한 리소스만 접근할 수 있는 정책을 부여했습니다. 예를 들어, DB 운영자는 EC2나 S3에 접근할 필요가 없으므로 RDS 관련 권한만 주는 식입니다. 또 액세스 키는 주기적으로 로테이션하고, MFA를 통해 보안을 강화했습니다.”

---

### **5. 장애가 발생했을 때 어떻게 대응하시나요?**

  

**답변:**

“우선 모니터링 알람을 통해 장애 상황을 빠르게 인지합니다. 서비스 로그와 자원 상태를 점검한 뒤, 장애가 애플리케이션 문제인지 인프라 문제인지 구분합니다. 이전에는 VM 생성 과정에서 병목이 발생한 적이 있었는데, 이벤트 기반 아키텍처로 개선해 장애가 확산되지 않도록 한 경험이 있습니다. AWS에서도 동일하게 서비스와 인프라를 분리해 대응할 수 있다고 생각합니다.”
